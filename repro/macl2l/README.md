# macl2l correction package — Lastprognose-Challenge SoSe26

This package documents the **2026-07-06 correction** of the `macl2l` and
`macl2l_entsoe` leaderboard submissions: why they were regenerated, exactly how,
and how to reproduce/verify the corrected files.

## What happened

All live submissions of `macl2l` (targets 2026-06-25..2026-07-06) and
`macl2l_entsoe` (2026-06-22..2026-07-06) were produced with a defective
completion of the partial current day: the submitter copy-filled the missing
D-1 evening hours with the value one week earlier (t−168), which silently made
the model's level anchor LAST WEEK's load. During the late-June heat ramp-up and
the early-July drop the forecasts became D-7 level clones (corr(daily error,
D7−actual) = 0.98/0.99), degrading `macl2l` to a mean MAE of ~2940 MW.

Root cause analysis and fix: KB entry
`tbb-skills4agents/plugins/spot-knowledge/knowledge/macl2l/2026-07-06-partial-day-fill-level-bias-fix.md`.
The fix (a leakage-clean learned Ridge completion of the partial day) shipped in
`bart26l-vorlesung@95cf920` and is live from the 2026-07-07 submissions onward.

## Correction policy (what the corrected files are and are not)

- **Replay standard: day-ahead complete D-1.** Each corrected forecast for target
  day D was regenerated by the stock fixed submitter with `--target-day D
  --skip-download` against a complete data cache: it uses only data strictly
  before D 00:00 UTC (origin-anchored, leakage-clean by the day-ahead standard).
  This is *slightly more favorable* than the original operational conditions
  (submissions were made ~17–19 UTC on D-1 with the day still partial) — under
  the original conditions the fixed learned completion scores ~1503 MW at a
  20:00 UTC cutoff vs 1426 MW with complete D-1 (replay measurements in the KB
  entry above).
- **Checkpoints: the current ones for all days.** `macl2l` days before 2026-07-02
  were originally submitted from an older checkpoint; the correction replays all
  days with the current robust-target checkpoint (2026-07-01). For
  `macl2l_entsoe` the checkpoint has been unchanged since 2026-06-21, so replay
  checkpoint = original checkpoint for every corrected day.
- **All replays are used, even where worse.** The corrected record is purely the
  fixed pipeline's output — no per-day selection against ground truth (that
  would be hindsight cherry-picking). Two days score worse than their originals
  and are kept anyway: `macl2l` 2026-06-30 (1335.1 → 2577.5) and
  `macl2l_entsoe` 2026-06-30 (1201.9 → 2296.6).
- The entsoe median-guard sidecar was reset and rebuilt chronologically during
  the replay (oldest day first), so each replayed day's adaptive guard threshold
  used only deviations from earlier days — no future leakage.

## Reproducing the corrected files

Requires the private repos `bart26l-vorlesung` (submitter) and `macl2l` (model +
checkpoints) at the commits pinned in `MANIFEST.md`, and an ENTSO-E cache
covering 2023-05..2026-07-07 under `~/spotforecast2_data/ddmo_macl2l/interim/`.

```sh
cd ~/workspace/bart26l-vorlesung
# entsoe, chronologically (guard sidecar reset to header-only first):
printf 'date,dev_mw\n' > ~/spotforecast2_data/ddmo_macl2l/median_guard_devs.csv
for D in 2026-06-{22..30} 2026-07-0{1..6}; do
  uv run python scripts/macl2l.py --variant entsoe \
    --macl2l-root ~/workspace/macl2l/src --skip-download --target-day "$D"
done
# nofc:
for D in 2026-06-{25..30} 2026-07-0{1..6}; do
  uv run python scripts/macl2l.py --variant nofc \
    --macl2l-root ~/workspace/macl2l/src --skip-download --target-day "$D"
done
```

This is the **operational profile** (matches what was submitted to the
leaderboard, given the same cache snapshot); MLX float accumulation across
machines may differ in the last decimals, so verify with tolerance rather than
bit-exactness. The exact submitted bytes are pinned in `expected/SHA256SUMS`.
